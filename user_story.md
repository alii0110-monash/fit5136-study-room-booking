## 1. User Stories and Acceptance Criteria
### Feature 1: Customer Registration, Login, and Account Management
#### User Story 1.1
作为一名蒙纳士大学的学生，
我想要使用我的学号、邮箱和密码注册一个账户，
以便于我可以登录系统并获得一笔初始体验金来预订自习室。

**Acceptance Criteria**
1. 终端提示用户输入学号(必须为纯数字)、合法的邮箱格式和密码(不能为空且长度大于6位)。
2. 如果学号已被注册，系统必须提示错误信息并要求重新输入。
3. 注册成功后，终端显示绿色的“注册成功，已为您发放 50 积分初始体验金”提示。
4. 系统必须将该注册信息成功持久化保存到文本数据文件(如 accounts.csv )中，确保数据在重启后依然有效，且满足不使用数据库的约束。

#### User Story 1.2
作为一名已登录的学生，
我想要在主菜单中查看我的个人资料和活动历史记录，
以便于我可以了解我的当前余额和过去使用过的房间记录。

**Acceptance Criteria**
1. 用户在主菜单输入指定数字选择“查看账户”。
2. 系统清屏并打印出用户的学号、邮箱和当前可用余额。
3. 系统在下方以文本表格形式列出该用户历史已完成的预订记录(含日期、时间、房间号)。
4. 系统在屏幕底部提供“按 0 返回主菜单”的选项，确保控制流的闭环。

### Feature 2: Room Management
#### User Story 2.1
作为一名系统管理员，
我想要在系统中添加一个新房间，并选择性地配置额外设备，
以便于为学生提供多样化的自习室选择。

**Acceptance Criteria**
1. 管理员输入基本的房间号、最大容量(必须为大于0的整数)和基础时租价格(必须为正数)。
2. 系统弹出确认提示:“是否需要为该房间配置可选设备?(Y/N)”。
3. 若选择 Y，系统循环提示输入设备名称(如白板)及附加费用，直到管理员输入“完成”。
4. 系统必须将新房间的完整信息(包含设备列表)成功持久化保存到文本数据文件中，确保数据在重启后依然有效。

#### User Story 2.2
作为一名系统管理员，
我想要将某个存在设施损坏的房间从系统中移除，
以便于防止学生预订到无法使用的房间。

**Acceptance Criteria**
1. 管理员在菜单中选择“删除房间”，并输入目标房间号。
2. 如果该房间当前有未完成的“未来预订”，系统必须拒绝删除，并提示“该房间已有排期，无法直接删除”。
3. 如果房间满足删除条件，系统需请求二次确认。
4. 确认后，系统必须将该更新状态成功持久化保存到文本数据文件中。

### Feature 3: Booking Management
#### User Story 3.1
作为一名需要自习的学生，
我想要根据日期、特定时间段和最低人数容量来筛选房间，
以便于我能快速找到满足我团队开会需求的空闲房间。

**Acceptance Criteria**
1. 系统提示用户依次输入:目标日期(不能为过去的日期)、开始时间、结束时间(结束时间必须晚于开始时间)、最低人数。
2. 系统需过滤掉容量不满足最低人数要求的房间。
3. 系统需读取现有的预订记录，过滤掉在该日期和时间段内已经有冲突的房间。
4. 系统将满足所有条件的可用房间列表打印在终端屏幕上，并附带对应的序号供后续选择。

#### User Story 3.2
作为一名已找到合适房间的学生，
我想要确认预订该房间，
以便于为我保留该时段的使用权。

**Acceptance Criteria**
1. 用户在筛选结果列表中输入对应房间的序号。
2. 系统展示该次预订的确认摘要(包含房间号、预订时段、单价、预计总价)。
3. 在用户最终确认扣款前，系统必须进行最后一次可用性二次校验 (Double-check)。如果该房间在用户犹豫期间已被其他人预订，系统需拦截并提示错误，以防止重复预订或时间安排冲突。
4. 校验通过后，系统将预订记录(状态为待支付)成功持久化保存到文本数据文件中。

### Feature 4: Checkout and Booking Handling
#### User Story 4.1
作为一名刚提交预订请求的学生，
我想要使用我账户中的可用积分进行结账，
以便于完成预订流程并锁定房间。

**Acceptance Criteria**
1. 系统检查用户的当前账户积分是否大于等于本次预订的账单金额。
2. 如果余额不足，系统提示“余额不足，结账失败”，并允许用户返回主菜单。
3. 如果余额充足，系统显示扣款成功信息。
4. 系统必须在文本数据文件中扣除对应积分，并将该订单状态更新为已确认。

#### User Story 4.2
作为一名需要长时间自习的学生，
我想要在预订超过 4 小时的时候自动享受“Deal Package”折扣，
以便于我能以更划算的价格使用房间。

**Acceptance Criteria**
1. 在生成结账账单时，系统自动计算预订的时长。
2. 如果预订时长达到或超过 4 小时，系统在账单中明示:“已触发长时间学习优惠套餐 (Deal Package)，总价享受 8 折优惠”。
3. 扣款时，系统必须按照打折后的金额从学生账户中扣除积分。
4. 结账完成后，系统提示支付成功，并提供返回主菜单的选项。
