# UI Prototype & Presentation 团队展示任务分配与统筹指南(终稿)
## 1. 核心规则与行政合规要求 (The "Must-Dos")
- **时间红线**：整个演示过程绝对不能超过10分钟。超时可能会被打断并影响评分。
- **文件格式与命名**：最终提交到系统的文件必须是PPT 或PPTX格式。文件必须严格命名为 FIT5136_S1_2026_Appliedxx_TeamX_A1-presentation.pptx。
- **原型链接展示**：千万不要忘记在幻灯片的显眼位置(如单独一页)附上完整的 LucidChart 共享链接。
- **[关键合规] AI 使用声明统筹**：PPT 负责人在最终组装打包提交文件时，必须统筹团队将一份名为 Foundation Models Used 的 AI 声明文件(txt, pdf 或 word 格式均可)连同全员签名的 Cover Sheet 一起放入提交流程中。不交会被视为学术不端。

## 2. 人员调配与排练统筹 (Run-sheet & Rehearsal Director)
### [关键更新] 明确“全员参与”的剧本与时间切分 (Run-sheet)
如果有成员缺席演示，该缺席成员的此项得分将直接计为0分。PPT 负责人必须制定一份精确到秒的 5 人演讲时间表。

**建议时间分配**：
1. 开场概览与高层逻辑流(成员 1，约 2 分钟)
2. 核心 User Story 讲解 1 & 2(成员 2 & 3，约 3 分钟)
3. 核心 User Story 讲解 3 及 UI 原型演示(成员 4，约 2 分钟)
4. 可用性原则落地与总结(成员 5，约 1.5 分钟)
5. 预留 Q&A 缓冲时间(约 1.5 分钟)

## 3. PPT 必须涵盖的四大核心内容 (Content Checklist)
在短短10分钟内，团队必须向导师展示以下四大板块：
1. **全局概览与需求对齐**：展示数字原型、应用程序的高级事件流(High-level flow)，并清晰解释该设计是如何满足客户(TA)最初提出来的各项需求的。
2. **[关键更新] 3 个核心 User Story 深挖与挑选策略**
    - 挑选要求：从撰写的8个User Stories 中挑选出最引以为傲的3个，且必须体现系统的多样性。
    - 展示要求：不仅要展示这3个故事及其验收标准 (Acceptance Criteria)，还要直接对应到画好的UI界面进行讲解。
3. **可用性原则的落地**：必须明确展示原型是如何体现选定的可用性设计原则的。重点讲解如何把 Ben Shneiderman 的三条原则(反馈、减少记忆负担、允许撤销)运用在了纯文本的终端界面上。
4. **导师答疑 (Q&A) 准备**：演示环节中，导师(TA)会针对原型和设计思路提出问题。讲演者必须对项目的“每一个层面(every aspect)”都有极其深入的了解，即便负责画图排版的同学，也要对底层 CSV 持久化逻辑对答如流。

## 4. 评分标准(Rubric)中的“高分密码” (How to get HD)
**兼顾无障碍的视觉质量 (Accessibility in mind)**：PPT不仅要结构清晰、视觉效果出色，还要考虑“无障碍性”。排版不能过于拥挤，字体必须足够大，且必须使用高对比度的配色(比如黑底白字或白底黑字)，确保最后一排的观众也能轻松看清。
